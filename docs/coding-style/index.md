## 文件组织

- 一个文件对应一个逻辑组件 （比如：资源类型，资源加载器）
- 自动生成的文件路径应当包含 `gen` 标识符，不要手动改它
  - `gen/` 目录下的文件是生成的
  - `*.gen.*` 后缀的文件是生成的

## 代码格式

- **缩进**: 使用 Tab 缩进，禁止使用空格缩进
- **换行符**: 使用 LF（Unix 风格）
- **编码**: 使用 UTF-8 编码保存文件

## 易犯错点与建议

- 涉及 `null` / `undefined` 写法、可选参数、可选链或三目运算时，优先查阅 [TypeScript 代码规范](./TypeScript.md)
- 涉及函数作用域、抽象边界、参数设计和、字符串输入、fallback 写法时，尤其是新增代码实现前优先查阅 [实现细则](./Implementation.md)
- 牢记这条边界原则：校验、规整、纠错只应发生在最靠近数据来源的边界；一旦数据已经被 TypeScript、XSchema 或协议约束收口，后续控制器、服务层、领域层必须直接信任它，不得重复补偿上游
  - 不要在下游重复写 fallback、强转、trim、空值揉平、默认值兜底
  - 如果值非法，应优先修正上游边界，或显式报错/拒绝，而不是偷偷改写后继续执行
  - 展示层允许做展示语义的 fallback，但这不属于数据规整
  - 具体边界划分、判断顺序与字符串类典型反例，见 [实现细则](./Implementation.md)
- 调用点尽量薄、接口自己处理空值、输出结构贴近真实消费需求；不要为了“喂参数”或“结构统一”在调用点额外铺分支或包单字段对象
- 空值语义默认统一为 `undefined`；参数可选就用 `?`，不要顺手引入 `| null`，调用点不要先写三目分支喂参数，细则见 [边界与补偿](./Implementation.md)
- 牢记这条判断标准：局部辅助逻辑的作用域要贴着实际使用点放，不能为了“先定义好”就提前提升到外层初始化范围
  - 单次使用且只服务于某个回调、分支或局部流程的辅助函数/变量，必须放在离调用点最近的函数作用域内
  - 不要把仅被某一个内层回调使用的局部逻辑提前提升到外层初始化范围、模块级作用域或兄弟流程共享范围
  - 只有在同一文件内存在明确复用、需要形成稳定命名边界，或确实被多个流程共同依赖时，才允许提升作用域
- 三目运算符 (Agent 最易犯错点之一)
  - 如果分支逻辑较复杂或包含多个操作，亦或是一行写不下时不允许使用三目运算符
  - 当分支逻辑简单且表达清晰时可以使用三目运算符；
  - 如果需要换行或是两个以上条件综合判断， 那就不要使用三目运算符
