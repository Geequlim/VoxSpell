# AGENTS.md

此文件为 AI 编码代理的工作指南，聚焦代码编辑与生成。 适用于前端、后端、测试、工具链相关的代码与文档编辑任务。

## 开工前清单

- 凡写代码前, **务必阅读 [编码规范](./docs/coding-style/index.md) 和相关子章节**，确保代码风格、类型使用、函数设计等符合项目要求
- 凡涉及代码编写、审查、重构或测试，务必先阅读并遵守 [karpathy-guidelines](.agents/skills/karpathy-guidelines/SKILL.md)，将其作为默认编码行为准则
- 在修改代码或文档前，先判断当前是否处于讨论阶段；若用户尚未明确要求执行，不得修改文件
- 理解项目架构与代码组织，特别是逐级从属设计模式，查阅 [架构文档](./docs/architecture/index.md)
- 根据任务需要，主动阅读技能文档中引用的相关资料，理解相关技术细节、设计原则与最佳实践

### 任务相关资料查阅

- 写文档前，先阅读 [tiny-docs 技能](.agents/skills/tiny-docs/SKILL.md)

## 讨论与执行边界

- 当用户在讨论问题、询问原因、请求解释、评估方案或表达期望时，不应直接修改代码、创建文件、运行会改变工作区的命令或开始实现。
- 讨论阶段可以主动进行探索，包括读取文件、搜索代码、查看 diff、检查类型定义、查看配置、运行不会改变工作区的只读命令，或在 `/tmp/`、`.temp/`、`.cache/` 等临时位置编写一次性验证脚本，用于理解上下文、定位问题并给出更可靠的解释和方案。
- 讨论阶段不得修改仓库源码、创建正式文件、安装依赖、变更配置、格式化代码或启动实现流程；临时验证产物不得散落到源码目录，任务结束后如不再需要应清理。
- Agent 应当先给出问题解释、解决方案、影响范围、风险与验证方式。
- 只有在用户明确表达执行意图后，才能开始修改代码。明确执行意图包括：
  - “按这个方案改”
  - “开始实现”
  - “帮我修改”
  - “落地到代码”
  - “写入 AGENTS.md”
  - “写入指定文件”
  - “按上述方案修改文件”
- 如果用户表达含糊，例如“这样可以吗”“应该怎么做”“我希望以后这样”，应默认处于讨论阶段，不应动代码。
- 如果方案尚未达成共识，应继续讨论而不是实现。
- 开始修改前，应复述将要执行的明确方案和目标文件；如果用户已经明确要求执行，复述后可以开始修改；如果用户只是讨论到一个方案，复述后必须等待用户确认。

### 暂存区审查隔离

- 当用户要求查看、分析或审查 staged / 暂存区变更时，Agent 默认不得修改 git index，也不得自动 stage 自己的修改；只有用户明确要求加入暂存区时才允许更新 index。
- 暂存区审查后如需继续修改，Agent 的新增修改必须默认留在工作区未暂存状态；若修改已存在 staged 变更的同一文件，任务结束时必须说明这些交叉文件。

## 默认编码准则

- 在进行代码编写、代码审查、重构、修复 bug、测试编写或文档化技术方案时，必须遵守 [karpathy-guidelines](.agents/skills/karpathy-guidelines/SKILL.md)
- 在开始修改代码前，应先按该技能明确假设、范围、成功标准和验证方式
- 实现时应优先选择最简单可行方案，只修改与用户请求直接相关的代码
- 完成后应按任务目标执行对应验证，不能只凭主观判断认为完成

## 提交前检查与验证策略

- 完成修改后，先进行自查，包括实现逻辑、类型正确性、代码风格、影响范围与验证结果。
- 完成修改后，在仓库根目录执行固定命令 `yarn tiny fix-worktree`，用于确保当前工作区的代码风格统一；不要向该命令追加文件路径参数。
  - 仅修正代码风格，不影响测试结果和编译结果；如果测试或编译已在此之前通过，执行后不要重复跑。
- 除固定 lint/fix 外，根据改动类型选择额外验证，避免机械执行无关检查。

## 关键约定

### 代码风格与质量红线

- 在 TypeScript 和 C++ 文件中永远使用 tab 缩进，禁止空格缩进，使用 lf 换行
- 修改代码前必须确认已阅读与任务相关的编码规范、架构文档和技能文档。
- 修改代码后必须按「提交前检查与验证策略」完成自查和验证。
- 此项目的开发者非常重视代码质量和一致性，遵守这些约定有助于维护项目的可维护性、可读性和稳定性，同时也能提升开发效率和团队协作效果
- 严重警告：本项目无法容忍不符合编码规范的代码。实现必须严格遵守 [编码规范](./docs/coding-style/index.md) 及相关子章节；粗糙实现、随意重构、跳过规范阅读、跳过自查或未经验证就声称完成，都视为严重质量问题。

### 工具链与依赖

- 项目使用 yarn v4, 不要尝试使用 npm 或其他包管理工具
- 使用 [tiny 命令行工具](./docs/tools/tiny-cli.md) 执行 project.tiny 中定义的快捷指令，而不是使用 npm script
- 涉及依赖库的变更（添加/删除/升级）时，不论对子模块或根项目的 `package.json` 修改，都需要在根目录执行 `yarn install` 来确保 `yarn.lock` 正确更新

### 临时文件

- 如果需要创建临时文件、临时目录、调试输出或一次性脚本产物，应优先使用系统临时目录（如 `/tmp/`）或仓库内约定的 `.temp/`、`.cache/` 目录
  - 不要把临时文件散落到源码目录、模块目录、`tests/` 目录或应用目录中
  - 任务结束后，如这些临时产物不再需要，应及时清理

### 输出格式

- 输出文件路径时，路径与前后文之间一定要添加明确的空格，避免渲染工具将前后文与路径混淆成一个整体导致无法正确识别
  - 错误示例: `位置：index.ts:402。存在异常`, 路径前后没有空格，直接和中文符号连接
  - 正确示例: `位置: index.ts:402 存在异常`, 使用英文冒号并路径前后添加空格，确保渲染工具正确识别路径部分

## 限制与禁止

- 不手工编辑 `yarn.lock` 文件；依赖变更只能通过根目录 `yarn install` 让其自动更新
- 不直接操作数据库或外部服务
- 除非用户明确要求，否则 agent 不应介绍、选择或执行任何发布流程。
